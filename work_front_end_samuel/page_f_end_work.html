ENGENHARIA DE REQUISITOS

Definições de Engenharia de Requisitos

- Para Pfleeger (2004), um requisito é uma característica do sistema ou a descrição de algo que o sistema é capaz de realizar para atingir os seus objetivos;

- No SWEBOK (2004), um requisito é descrito como uma propriedade que o software deve exibir para resolver algum problema no mundo real;

- Segundo Nuseibeh e Easterbrook (2000) requisitos são expressões das necessidades de um stakeholder para atingir objetivos particulares;

Ou seja, a Engenharia de Requisitos é uma disciplina que se concentra em definir, documentar e manter os requisitos de um sistema de software. É nesse ponto que as necessidades e desejos do cliente são traduzidos em um roteiro detalhado para a criação de um produto que atenda às expectativas, supere obstáculos técnicos e cumpra regulamentações pertinentes.

Este processo não é apenas sobre identificar o que o software deve fazer, mas entender como essas funcionalidades se alinham com os objetivos gerais do projeto, a experiência do usuário final e os padrões de qualidade do setor.

Durante o estudo surgir algumas perguntas como: Por que é difícil compreender o que o cliente realmente deseja? Ou ainda: O cliente não deveria saber o que é necessário? Os usuários finais não deveriam entender as características e funções que trazem benefícios?
E surpreendentemente, em muitos casos a resposta a essas perguntas é "NÃO". Mesmo que os clientes e usuários finais expressem claramente suas necessidades, elas podem mudar ao longo do projeto. É a fase mais crítica e sujeito a erros no desenvolvimento de software.

Níveis de Requisitos

Os níveis de requisitos na Engenharia de Requisitos definem o grau de abstração e detalhamento das necessidades de um sistema. Eles organizam as informações de forma progressiva, partindo de uma visão mais geral do negócio até especificações técnicas mais detalhadas, garantindo clareza na comunicação entre clientes, usuários e desenvolvedores.

Dividindo-se principalmente em:

  • Requisito de Usuário: Declaração, em linguagem natural com diagramas simples (como tabelas), dos serviços esperados e suas restrições.

    Exemplo: Um campo para a data e hora do cadastro e uma opção para selecionar o estado civil.

  • Requisito de Sistema: Declaração detalhada e estruturada dos requisitos do usuário, direcionada à equipe de desenvolvimento. Apresenta especificações funcionais e não funcionais, servindo como base formal para implementação e validação do sistema, funcionando também como um contrato técnico entre cliente e desenvolvedor.

    Exemplo: A tela de cadastro incluirá campos de data e hora preenchidos automaticamente e uma seleção de estado civil: casado, solteiro ou separado.

  • Requisito de Projeto (Especificação de Projeto): Corresponde à definição técnica do software, incluindo arquitetura, modelagem, padrões de desenvolvimento, tecnologias utilizadas e critérios de desempenho. É o nível mais detalhado, voltado para orientar a construção efetiva do sistema.

    Exemplo: O sistema deve processar pagamentos via PIX em até 3 segundos, garantindo disponibilidade de 99,9%.

Concluindo: Os níveis de requisitos organizam as necessidades do projeto de forma estruturada, indo dos objetivos estratégicos até os detalhes técnicos do sistema. Essa divisão facilita a comunicação, reduz erros e garante que o software desenvolvido esteja alinhado com as necessidades do negócio e dos usuários. Assim, entender esses níveis é fundamental para o sucesso de qualquer projeto de software.

Tipos de Requisitos

Os tipos de requisitos classificam as diferentes características que um sistema deve atender. Eles ajudam a organizar as funcionalidades, as qualidades esperadas e as regras específicas do contexto onde o software será aplicado. Essa divisão facilita a especificação correta do sistema e garante que todos os aspectos importantes sejam considerados durante o desenvolvimento.

São eles:

  • Requisitos Funcionais (RF): São aqueles que definem as funções que o sistema ou um componente deve realizar, ou seja, suas entradas, saídas e comportamentos. Eles representam uma solicitação de funcionalidade que o software deve implementar. É uma exigência, necessidade ou desejo que será materializado no sistema

    Exemplos: Cadastrar clientes; Emitir relatórios; Permitir login de usuário; Alterar o estado de um cliente.
  • Requisitos Não Funcionais (NF): Os requisitos não funcionais tratam das restrições e atributos de qualidade do sistema. Eles definem como o sistema deve se comportar em termos de desempenho, segurança, confiabilidade, usabilidade, portabilidade e padrões técnicos. Sempre que possível, devem ser descritos de forma mensurável para permitir testes objetivos.

    Exemplos: O sistema deve responder em até 5 segundos; Deve ser protegido contra acessos não autorizados; Deve funcionar em diferentes configurações de hardware.
  • Requisitos de Domínio (ou Negócio): Os requisitos de domínio são derivados da área de aplicação do sistema. Eles representam regras específicas do negócio, fórmulas, normas internas ou legislações que precisam ser respeitadas.

    Exemplos: Cálculo de imposto conforme regra fiscal; Comissão de vendedores definida em percentual fixo; Cálculo de média com fórmula específica.

Concluindo: Os tipos de requisitos permitem organizar de forma clara tudo aquilo que o sistema deve atender. Os requisitos funcionais definem as ações e serviços do software, os não funcionais estabelecem padrões de qualidade e restrições, e os requisitos de domínio garantem que o sistema respeite as regras específicas do contexto onde será aplicado. Essa classificação é fundamental para evitar ambiguidades, facilitar a validação e garantir que o sistema atenda corretamente às necessidades do negócio e dos usuários.

Stakeholders

Stakeholders são as partes interessadas em um projeto ou sistema. São indivíduos ou organizações que podem ser afetados direta ou indiretamente pelo desenvolvimento e uso do software.

No contexto de Engenharia de Software, stakeholders não são apenas clientes. Qualquer pessoa ou entidade que tenha interesse, influência ou impacto no projeto pode ser considerada stakeholder.

Quem pode ser STAKEHOLDER?

  • Colaboradores;
  • Clientes;
  • Parceiros comerciais;
  • Fornecedores;
  • Concorrentes;
  • Investidores;
  • Gestores;
  • Governo;

Por que STAKEHOLDERS são importantes?

Os requisitos do sistema surgem das necessidades dos stakeholders. Cada grupo pode ter expectativas diferentes:

  • O cliente quer que o sistema resolva seu problema;
  • O usuário quer facilidade de uso;
  • O gestor quer controle e relatórios;
  • O governo pode exigir conformidade com normas e leis;

Ignorar algum desses grupos pode gerar falhas nos requisitos e problemas no projeto.

Concluindo: Identificar corretamente os stakeholders é essencial na Engenharia de Requisitos, pois são eles que definem as necessidades que o sistema deve atender. Compreender seus interesses ajuda a reduzir conflitos, evitar retrabalho e garantir que o software entregue valor real a todos os envolvidos.
